# codex-web 内核升级与线上问题处理总结

本文档汇总本轮工作的完整过程：Codex Desktop 内核从 26.707.61608 升级到 26.715.31925 的实施方法、升级后暴露的几个开发问题（含文件拖拽上传失败）的排查与修复思路、以及升级之外顺带处理的两起线上问题（移动端导航栏错位、后端"假死"）。目的是把方法沉淀下来，方便下次同类工作直接复用判断依据，而不是重新摸索。

## 一、任务的起点：为什么重新拟定计划

此前有另一个 AI 给出过一版升级计划，被拒绝，原因很明确：内核版本号判断和兼容适配细节不够扎实，尤其是"浏览器模式下拖拽上传图片经常失败"这一类细节问题没有被覆盖。这次没有直接复用旧计划，而是重新走了一遍最笨也最可靠的路径——把新版本的 Desktop 安装包和 CLI 二进制都实际下载下来解包，与仓库里当前基线逐文件 diff，而不是只看 npm dist-tag 或 changelog 里写的版本号。

这一步就纠正了两个此前被误判的事实。第一，CLI 的目标版本从 npm 标注的 latest（0.144.6）纠正为二进制里实际内嵌的版本字符串 0.145.0-alpha.18——如果只看 dist-tag 会升错版本。第二，fork 里 `scripts/prepare_asar` 定义的 14 个补丁在旧内核上其实全部处于生效状态，此前的评估误判为"5 个被禁用、9 个生效"。这两点如果不核实就直接照抄别人的计划，后续所有补丁重定位工作都会建立在错误的基线上。

## 二、升级本身怎么做：diff 驱动的补丁重定位

新旧两个版本的 webview 资源文件名都带有内容哈希，同一段逻辑在新版本里会出现在文件名完全不同的 chunk 里，行号和字节偏移也全部漂移。直接照搬旧 patch 文件的 `@@` 行号去 patch 新文件基本不可行，所以采用的方法是：对每一个补丁，先在旧文件里定位到它改动的那段代码的**特征字符串**（通常是一个不会被压缩打包工具重命名的字面量，比如某个 aria-label、某个错误提示文案、某个固定的 CSS 类名片段），再用这个特征字符串去新解包出来的资源目录里做全局检索，并且要求检索命中数量正好等于 1——如果命中 0 次或大于 1 次，说明这个特征字符串选得不够独特或者这段逻辑在新版本里已经被重构，不能贸然假设锚点位置，需要换一个更独特的特征串重新定位，或者放弃自动重定位、转为人工判断。

这套"先验证唯一命中，再动手改"的流程贯穿了全部 14 个补丁的重定位过程，其中 12 个补丁顺利在新版本里找到了唯一锚点并重新生成了补丁文件，套用 `patch --fuzz 3` 的容错度整体验证通过；`webview-thread-title` 和 `webview-prompt-search-param` 这两个补丁因为新版本里对应逻辑的锚点定位不够唯一、贸然套用有静默改错代码的风险，被判断为不值得冒险，选择继续保持禁用状态，把具体的候选锚点位置记录下来留作人工复核项，而不是硬着头皮打上去。

版本升级还牵涉到 `default.nix`、`nix/codex/default.nix`、`scripts/prepare` 里的版本号和资源包 SRI 哈希需要同步更新——这几处如果漏改，构建出来的包会和实际下载校验的哈希对不上，属于升级流程里容易漏掉但一旦漏掉会直接导致构建失败的环节。

整个升级流程严格限定在 `MittyLeeisOK/codex-web` 这个 fork 自己的仓库范围内，不去动上游 `0xcaff/codex-web` 的内容，这是升级开始前就明确的范围约束。

## 三、升级后暴露的开发问题与修复思路

补丁本身重定位成功、构建也能通过，不代表升级完成——升级之后在无头浏览器里做实际交互测试时，暴露出三个独立的运行时回归问题，这几个问题的共同点是：它们都不是补丁锚点本身的问题，而是新版内核代码调用了一些这个 fork 里从未真正实现过的 Electron API，旧版本的代码路径恰好没有触发这些调用，新版本触发了，问题才暴露出来。

第一个问题出现在应用刚启动时，界面直接抛出错误边界崩溃页面，控制台报 `Fi.appUpdates?.setSparkleQueryParams is not a function`。定位到这是新内核里一个更新检查相关的模块调用了 `appUpdates.setSparkleQueryParams`，而这个 fork 的 Node 端只实现了一个精简的 `appUpdates` 对象、没有这个方法。修复方式是在 `webview-app-host-services.patch` 里给 `appUpdates` 对象补一个空操作的 fallback 方法，不影响其他字段。

第二个问题更隐蔽也影响面更大：Node 端的 `net.request()` 之前只是一个几乎什么都不做的桩实现，缺少 `getUploadProgress()`、`.on()`、`.end()`、`.abort()` 这些方法。新内核里有一条走带进度追踪的文件上传逻辑，会在一个 `setInterval` 定时器回调里调用 `getUploadProgress()`，这个方法不存在，直接抛出 `TypeError: u.getUploadProgress is not a function`，而这个异常发生在定时器回调里没有被上层捕获，会直接把整个 Node 进程内的 IPC 桥接搞挂——所有客户端的 IPC 连接因此表现为"卡在建立连接"。这也是为什么用户反馈里"IPC 建立不起来"和"拖拽上传图片失败"其实是同一个根因导致的两个症状。

修复思路是把 `net.request()` 从空桩换成一个基于 `undici`（`ProxyAgent`、`request`）的真实实现，支持流式上传请求体、走代理、并且能正确追踪上传进度、暴露 `.on()`/`.end()`/`.abort()` 接口，让新内核的调用方式能拿到它期望的返回值。这处改动之后又追着暴露出第三个问题：`net.isOnline` 同样缺失，导致 `remoteControl/status`、`thread/list` 这几个接口调用失败，一并在同一次改动里补上一个恒定返回 `true` 的 `isOnline()` 实现。

这三处修复都落在 `src/server/electron/index.ts`（以及编译产物 `.js`/`.d.ts`/对应 `.map` 文件）和 `webview-app-host-services.patch` 里，一次提交里完成，改动前用 `node --check` 确认编译产物语法正确，还单独写了一个独立的单元测试脚本，向本地临时起的一个 echo 服务器发送 500KB 的 POST 请求，验证新的 `net.request` 实现能正确返回 200、正确统计已发送字节数、进度回调数值正确，测试通过之后才认为这处改动是可信的。

## 四、拖拽上传图片问题是怎么解决的

这里需要澄清一个诊断过程中出现过的误判，避免以后重复走弯路。最初怀疑的方向是浏览器端代码里读取拖拽文件用的是 Electron 独有的 `window.electronBridge?.getPathForFile` / `File.path`，这两个 API 在真实浏览器里永远是 `undefined`，如果真是这个原因，需要新写一段浏览器端代码去读取 `DataTransfer.files`，再走已有的 `/__backend/upload` 上传接口补上这条链路。

但实际排查后发现，对话框里"拖拽图片附件"走的其实是另一条独立的代码路径——组件里读取拖拽文件用的是标准的 `DataTransfer.files`/`items` API，通过 `file.arrayBuffer()` 拿到内容后走 `fetch` 发起上传，这条路径本身没有依赖任何 Electron 独有接口，是完全符合浏览器标准的实现，此前怀疑的 `getPathForFile` 路径实际对应的是另一个功能（项目工作目录本地文件路径拖拽），跟"往对话里拖图片"不是同一件事，不需要为此新写浏览器端代码或新增补丁。

真正卡住拖拽上传的，就是上一节说的 `net.request` 崩溃：拖拽上传发起后，进度追踪定时器一调用 `getUploadProgress()` 就抛异常，把整个 IPC 桥接打挂，前端表现为上传卡死、后续操作全部无响应。修完 `net.request` 之后重新在无头浏览器里做拖拽上传测试，结果是拖拽事件正常触发、图片正常显示为已上传的附件（2 张图片、3 个附件元素）、页面控制台没有报错、进程日志里也没有再出现崩溃事件。也就是说，这个问题最终是作为服务端网络桩修复的直接结果被解决的，没有改动任何浏览器端的拖拽处理代码。

## 五、移动端导航栏错位问题

用户反馈左侧导航切换栏（收起/后退/前进按钮组）在桌面端看起来有点偏移、在移动端干脆看不见。这个问题一开始按"实际 APP 截图"去比对没有复现，后来被提醒应该去看浏览器里的实际渲染路径而不是内嵌的 Electron webview 截图，才换了方向：伪造真实 macOS Safari 的 User-Agent 字符串跑无头浏览器测试，稳定复现出问题——按钮组的横坐标从预期的 6～8px 跳到了 88px。

顺着这个 CSS 变量往回查，定位到问题出在一段负责计算 macOS 窗口"红绿灯"按钮避让边距的逻辑：这段逻辑判断当前是否运行在真正的 Electron 里，如果不是（`windowControlsOverlay` 为空、没有原生菜单），会退化到直接按 macOS 的 User-Agent 特征去套用红绿灯避让的预设边距值——但这个退化分支完全没有检查"现在到底是不是跑在真正的 Electron 桌面应用里"，只要 User-Agent 像 macOS，就会被当成桌面 Electron 应用处理，套上本不该存在的左侧留白。这也解释了移动端的另一个症状：整条祖先元素链被设置成了 `visibility: hidden`，只留下一个重复的"幽灯"元素挤在别的位置。

修复方式是给这段 macOS 特征退化逻辑加一道前置判断：只有当 `window.__ELECTRON_SHIM__` 这个标记存在时才允许套用桌面端的边距计算，这个标记是这个 fork 自己的浏览器端桥接代码注入的，只在通过 codex-web 自身的浏览器/webview 服务访问时才存在，真正的 Electron 桌面应用里永远不会有这个变量。加上这道判断之后，用同样的伪造 UA 重新测试，按钮组回到了预期的 8px 位置，移动端也不再出现幽灵元素，改动前后用截图做了对比验证。

## 六、后端"假死"问题：和补丁升级无关的独立故障

这是升级工作完成、代码推送之后新发现的一起问题，和内核升级没有直接关系，值得单独记录，因为它暴露的是运维层面而不是代码层面的坑。用户反馈后端好像挂了，实际排查发现进程本身一直健康运行、没有崩溃重启记录，真正的问题是某一次 IPC 请求（`thread/read`）被阻塞了 340759 毫秒（接近 6 分钟）。

顺着服务日志往前找，紧挨着这次超长阻塞之前的是一串 `git status`/`git ls-files` 子进程调用，每一个都在 60 秒超时后失败。根因是：这台机器上有 20 个已存在的 Codex 会话线程，它们的工作目录被记录为 `/root`（这是升级到"默认工作目录改为 `~/CodexProjects`"这个改动之前创建的旧会话，没有被迁移），而 `/root` 本身就是一个 git 仓库。这个仓库的 `.gitignore` 里没有排除掉几个挂载点目录——一个 61G 的 SSHFS 远程存储挂载、一个基于 rclone 的 OneDrive 挂载、以及几个体积较大的目录。Codex 自带的后台 git 状态/审阅监听逻辑每次扫描工作目录时都会尝试遍历这些挂载点，每一次遍历都要走网络 I/O，单次就能超时 60 秒，多次排队叠加就是用户实际感受到的"卡死"。

修复很简单：把这几个挂载点目录加进 `/root/.gitignore` 排除掉。改完之后同样的 `git status`/`git ls-files` 调用从原来的 60 秒超时变成几毫秒内返回，后端接口响应也从秒级恢复到个位数毫秒。这处改动只提交在 `/root` 这个独立仓库里，跟 `codex-web` 项目仓库是两回事，没有推送到任何远程。

这起问题的经验是：当一个后台服务的工作目录被设置成某个宿主机目录时，这个目录本身的 git 仓库卫生状况（有没有把网络挂载点排除干净）会直接影响这个服务的响应延迟，即便代码逻辑完全正常。排查这类"进程活着但响应慢"的问题，优先看进程本身的子进程列表和最近的系统日志里有没有长耗时或超时的调用，而不是先怀疑代码或急着重启服务。

## 七、顺带发现的 rclone 共享文件夹失效

在排查后端假死问题、确认 OneDrive 挂载导致扫描变慢的过程中，额外发现挂载点本身的日志里有一条持续报错：`个人保管库` 这个顶层目录反复报 `invalidRequest: invalidResourceId: ObjectHandle is Invalid`，直接访问这个路径会返回 I/O 错误。查了一下这个目录的对象 ID，前缀和账号自己网盘里其他正常目录的 ID 前缀完全不一样，符合 OneDrive"共享项/快捷方式"的特征——也就是说这不是一个真正存在于这个账号自己网盘里的文件夹，而是一个指向别处共享内容的链接，这个链接目前已经失效（可能是分享权限被收回、源文件被移动或删除、或者分享链接本身过期）。这个问题出在 OneDrive/SharePoint 那一侧的共享关系上，不是本地挂载配置或 rclone 本身的问题，本地没有办法修复，需要用户在 OneDrive 网页端重新处理这个共享项。挂载服务本身除了这一条持续的错误日志之外运行正常，不影响其他目录的读写。

## 八、这轮工作沉淀下来的几条经验

判断一个升级目标版本号，要以实际下载解压出来的二进制里内嵌的版本字符串为准，而不是 npm dist-tag 或第三方 changelog 上写的版本号，两者经常对不上。

给压缩打包过的前端资源重新打补丁时，不要直接照搬旧版本 patch 文件的行号去套新版本文件，要先用特征字符串定位、并要求全局检索命中数量正好为 1，命中数量不为 1 的情况一律当作"锚点不可靠"处理，不强行套用，宁可留作人工复核项也不要冒着静默改错代码的风险硬上。

排查"某个功能在浏览器里不工作，在桌面客户端里正常"这类问题时，一定要真的用无头浏览器加真实的 User-Agent 去复现，而不是靠比对内嵌 webview 的截图去类比推断——桌面客户端的 Electron webview 和浏览器标签页是两个完全不同的运行环境，很多平台检测逻辑的差异只有在真实浏览器环境下才会显现。

补一个存根 API（比如这次的 `net.request`/`net.isOnline`）的时候，要弄清楚新内核代码实际会调用到这个存根的哪些方法、以及在什么时机调用，只补上表面能编译过的签名是不够的，这次 `getUploadProgress` 就是运行时才会触发的调用，编译期完全检查不出来。

诊断出问题表面症状（比如"上传失败""IPC 卡住"）之后，不要急着假设两个症状对应两个独立修复点，先查这两个症状的调用链是否指向同一个根因——这次"拖拽上传图片失败"和"IPC 建立不起来"表面上是两件事，实际是同一处崩溃导致的两种表现。

线上服务"响应变慢但进程没崩"的问题，优先检查这个进程当前有没有耗时异常的子进程（比如卡住的 git 子进程），以及这个进程的工作目录本身是不是被无意中放在了一个包含网络挂载点、且没有做好 `.gitignore` 排除的目录里——这类问题的根因往往在运维层面而不是应用代码里。

发布前的验证顺序要固定：补丁应用无报错、编译产物 `node --check` 通过、构建脚本跑通、独立起一个进程做冒烟测试确认能正常握手响应，全部通过之后才重启线上服务，重启后再用无头浏览器把关键交互路径（含拖拽上传）重新跑一遍确认没有回归，而不是补丁打完就直接重启线上服务。

## 九、目前的状态

代码已经推送到 `MittyLeeisOK/codex-web` 的 `main` 分支，三次提交依次是内核升级本身、升级后回归修复、移动端 UI 错位修复，另有一次提交是记录本次升级过程的更新日志。生产服务已经用新代码重启并持续观察确认稳定。本地清理掉了升级过程中产生的临时对比目录、备份目录和无用中间文件，释放了约 5GB 磁盘空间，保留了必要的回滚备份。`webview-thread-title` 和 `webview-prompt-search-param` 这两个补丁仍然保持禁用状态，等待后续人工确认新版本里的准确锚点位置。`/root` 仓库的 `.gitignore` 修复解决了后端响应变慢的问题，但这台机器上还有 20 个旧会话线程的工作目录仍然指向 `/root`，如果之后想彻底避免同类问题，可以考虑把这些旧线程迁移到 `~/CodexProjects` 下。`个人保管库` 这个失效的 OneDrive 共享快捷方式还没有处理，需要用户自己在 OneDrive 网页端重新建立或移除这个共享项。
