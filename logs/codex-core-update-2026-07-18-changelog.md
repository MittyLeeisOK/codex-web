# Codex Desktop 内核升级与后续修复记录（2026-07-18 至 2026-07-19）

## 背景

上一轮由另一个 AI 提出的升级计划被拒绝，原因是缺少对内核版本改动本身的实际下载、解包与兼容性比对，也没有对浏览器模式下拖拽上传图片等已知问题给出验证方案。本轮工作按用户要求，只针对我们自己维护的 `MittyLeeisOK/codex-web` 分支进行，不处理上游 `0xcaff/codex-web` 的改动。全部工作均在本地完成，与 homepc 无关。

## 提交记录

三次提交已推送至 `MittyLeeisOK/codex-web` 的 `main` 分支：

`1c676c3` — Codex Desktop 内核从 26.707.61608 升级到 26.715.31925。工作内容包括更新 `default.nix` 与 `scripts/prepare` 中的版本号与哈希，并对新版打包结果重新定位 9 个原本生效的补丁锚点（新版对若干 webview chunk 做了重命名与合并，原补丁的行号与文件名均已失效）。另外重新映射并恢复启用了此前因锚点失效而被禁用的 3 个补丁：`webview-electron-shim-close-sidebar`、`webview-prosemirror-inputmode`、`webview-use-atfs-for-local-files`。两个补丁（`webview-thread-title`、`webview-prompt-search-param`）因新版本中对应代码位置的插槽数量无法确认，判断强行套用锚点风险较高，予以保留禁用状态，作为后续人工核实项。

`24a5891` — 修复升级后暴露的两个回归问题。其一，新版本每次页面加载都会无条件调用 `appUpdates.setSparkleQueryParams`，而项目自定义的 stub 对象缺少该方法，导致未捕获异常并使 `AppIntl` 错误边界在会话运行几小时后崩溃；修复方式是为该方法补充一个空操作兜底实现。其二，`src/server/electron/index.ts` 中的 `net.request()` 此前只是一个从不真正发起请求的占位实现，这在静默中破坏了文件上传进度轮询（`getUploadProgress` 未定义）及其他调用方；已替换为基于 `undici` 的真实实现，支持流式上传、进度事件、代理调度器与响应事件转译。同时补充了完全缺失的 `net.isOnline()`，此前其缺失导致 `remoteControl/status/read` 与 `thread/list` 请求失败。修复后经无头浏览器验证：无页面报错、无 `AppIntl` 崩溃，拖拽上传图片可正常完成且不再出现 `ERR_CONNECTION_RESET`。

`832212b` — 修复移动端/浏览器模式下侧边栏导航切换栏错位的 UI 问题（用户报告：切换栏歪斜且在移动端几乎不可见）。根因是 webview 中计算 `--spacing-token-safe-header-left/right` 的函数在检测到 macOS 用户代理时，无条件假设需要为 Electron 原生红绿灯窗口控件预留空间，即使当前运行在普通浏览器标签页、根本不存在这些控件。这导致侧边栏收起/后退/前进按钮组在任意 Mac 浏览器上向右偏移 82–92px，并在移动端 Safari 上出现部分裁切或与切换按钮重叠。修复方式是在该逻辑前增加一道判断：若 `window.__ELECTRON_SHIM__` 存在（该标记仅由本项目自身的浏览器/webview 服务端注入，真实 Electron 环境中不会出现），直接跳过用户代理探测分支，与已有的 `Ke()`（真实 Electron 且带原生菜单）判断保持一致的处理方式。

## 验证方式

三次修复均通过无头 Playwright 会话验证，覆盖桌面视口、窄屏视口与真实移动端 UA（含 iOS Safari 与伪装 macOS UA 的浏览器标签场景），并在生产服务重启后于日志中确认无残留报错、服务状态保持 `active`、HTTP 探测返回 200。UI 错位问题的验证额外用生产环境的真实登录会话（通过后台数据库直接签发一次性 1 小时有效期的会话令牌，测试结束后已从数据库中删除）复现了浏览器模式下的实际渲染效果，而非仅比对内嵌 Electron webview 的截图——这一区分是本轮排查中途才被用户点破的关键盲点。

## 经验与教训

排查侧边栏错位问题时，最初的假设方向完全错误：花费了大量时间去追踪一条看似与平台检测相关的变量重命名链（`Ke → xt → xl → lO`），最终发现这条链只是一个懒加载的会话状态 schema 模块，与窗口控件检测毫无关系。真正的检测逻辑藏在一个完全不同的对象结构（`cc`/`tc`/`rc`/`nc`）里，只是命名同样晦涩。这提示在压缩后的 bundle 中做代码考古时，同名或近义的变量链不能替代对调用点上下文的直接追踪——应当从触发问题的 CSS 变量或 DOM 属性反向定位赋值语句，而不是顺着看似相关的导入链正向猜测。

第二个教训是关于测试对象的选择：早期所有截图与行为验证都针对的是内嵌在 Electron 桌面应用中的 webview（通过 `localhost:8214` 直接访问），而用户报告的问题实际发生在浏览器标签页访问生产站点的场景下。这两者共享同一套前端 bundle，但运行时环境（是否存在 `window.__ELECTRON_SHIM__`、`navigator.windowControlsOverlay`、真实的操作系统与浏览器 UA）完全不同，足以让同一段代码走上不同的分支。日后遇到"仅在某平台/某入口方式下出现"的 UI 问题时，应当第一时间确认测试所用的访问路径与用户实际路径是否一致，而不是想当然地认为内嵌应用与浏览器版本行为等价。

第三，生产环境未配置面向自动化测试的登录后门，只能通过直接读取本地会话表结构、理解 `mitty_maccount_session` 与 `mitty_codex_gate` 两层 Cookie 的签发逻辑后，手工插入一条短期有效的会话记录来完成认证测试。这类一次性测试凭证应当在使用后立即从数据库中清除，本轮已经这样做，但更值得记录的是：如果这类真实浏览器路径的验证会反复用到，值得向后端提出一个明确的、范围受限的测试账号或测试 Cookie 签发机制，而不是每次都手工操作生产数据库。
